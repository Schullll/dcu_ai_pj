# 2026 인공지능 부트캠프 프로젝트
## 주제 : 중고폰 적정가 예측 서비스
### [설명]
중고폰의 조건(모델, 저장용량, 상태 등급)을 입력하면, 번개장터 실시간 API로 수집한 실거래 데이터(781건, 30개 기종)를 기반으로 학습한 
Scikit-learn 회귀 모델이 예측 중고가를 산출합니다. 
여기에 실제 중고폰 시장 벤치마크 기준 등급 보정(S/A/B/F)을 결합해 최종 예상가를 계산하며, GPT-5 mini(LLM)가 
이 결과에 대한 자연어 설명과 거래 체크리스트를 생성합니다..

### [주요 개선 과정 — 시행착오와 해결]

**1. 저장용량 변경 시 예측가 미변동 문제**
- 문제: 저장용량을 바꿔도 예측가가 거의 변하지 않음
- 원인: 가장 중요한 피처(신품가, 피처 중요도 58%)가 저장용량과 무관하게
  고정값으로 입력되고 있었음
- 해결: 20개 대표 모델의 실제 출고가를 조사하여, 모델별·저장용량별로
  정확한 신품가를 매핑표(phone_models.py)에 구축

**2. 저장용량별 가격 계산 오류**
- 문제: 저장용량 프리미엄을 반영하려고 normalized_new_price(로그 스케일 값)에
  배율(예: ×1.12)을 곱했더니 값이 비정상적으로 폭증
- 원인: 로그 스케일 값은 곱셈이 아니라 "덧셈"으로 보정해야 함(곱셈 시 지수적으로 부풀려짐)
- 해결: 아이폰X 실제 출고가(64GB 142만원 vs 256GB 163만원, 약 15% 차이)를
  검증 근거로 삼아, 20개 모델 전체의 실제 출고가를 조사(삼성/LG/애플 공식 자료,
  언론보도)한 뒤 ln(USD 환산가)로 재계산하여 정확한 가격 반영

**3. 사용기간(days_used) 반영 문제 — 가장 많은 시행착오**
- 3-1. 사용기간을 늘려도 모델 예측가가 불규칙하게 오르내림
      → 원인: days_used의 피처 중요도가 2.1%로 낮아 모델이 안정적 관계를 학습 못함
- 3-2. 후처리로 "연 12% 감가율"을 임의로 적용 → 그래도 순서가 뒤바뀜
      → 원인: 모델 예측값 자체에 이미 days_used가 반영되어 있어 후처리 보정과 이중 충돌
- 3-3. 모델 예측 시 days_used=0으로 고정 시도
      → 원인: 데이터셋의 실제 사용일수 범위가 91~1094일이라 0은 학습 범위 밖(외삽)이라
        비현실적으로 낮은 값 발생
- 3-4. days_used를 데이터 평균(674일)로 변경 → 그래도 신품가 대비 과도한 감가(약 83%↓) 확인
- 3-5. 실제 데이터의 "중고가/신품가 비율"을 검증
      → 최초 시도(단순 나눗셈)는 로그 스케일 값을 그대로 나눠 오류
      → exp(로그 차이)로 올바르게 재계산(real_retention_ratio)
- 3-6. 모델 예측값 기반으로 사용기간별 평균을 집계했으나 여전히 불안정
      → 근본 원인 발견: 이 데이터셋은 "동일 기기의 시간 흐름 추적"이 아니라
        "서로 다른 거래 건들의 독립적 스냅샷"이라, 모델이 감가 패턴을 학습할
        재료 자체가 부족함
      → 결론: 모델의 days_used 학습 결과는 신뢰하지 않기로 결정, 실측 데이터
        기반 접근으로 전환

**4. 표본 수 부족으로 인한 논리적 역전 발견**
- 문제: 실측 데이터를 200일 단위로 구간화했을 때, 0~200일(19건) 구간이
  200~400일(76건) 구간보다 가격유지율이 낮게 나오는 논리적 역전 발생
- 원인: 0~200일 구간 표본이 19건뿐이라 통계적으로 불안정
- 해결: 두 구간을 병합(0~400일, 95건)하여 표본을 확보, 이후 5개 구간
  모두에서 역전 없이 순감소하는 패턴 확인. 세분화의 정밀도와 통계적
  신뢰도 사이의 트레이드오프를 실제로 검증한 사례

**5. 계단식 감가율의 부자연스러움**
- 문제: 5개 구간을 계단식으로 그대로 적용하면, 같은 구간 안에서는
  사용일수를 바꿔도 예상가가 동일하게 나와 "예측 모델"로서 부자연스러움
- 해결: 5개 구간의 실측 평균값에 지수감가곡선(y = a·exp(-b·x))을
  scipy.curve_fit으로 피팅하여, 임의의 사용일수 입력에도 매끄럽게
  우하향하는 곡선 확보(전 구간 실측값과 오차 1%p 내외로 일치)

**6. 브랜드별 감가 속도 차이 검증**
- 배경: 실제 한국 중고폰 시장에서 잘 알려진 "애플의 중고가 방어력"이
  본 데이터셋(해외 중고시장 기반)에서도 나타나는지 검증
- 결과: 오히려 애플의 감가 속도가 삼성·LG보다 빠르게 나타남
  (예: 800~1000일 시점 애플 유지율 23.9% vs 3개 브랜드 통합 36.8%)
- 결론: "애플은 방어를 잘한다"는 특성은 한국 시장에 국한된 현상이며,
  데이터셋의 해외 시장 특성에는 반영되어 있지 않음을 확인. 브랜드별
  표본(8~12건)이 적어 통계적 신뢰도가 제한적이라, 잘못된 패턴을
  강화할 위험을 피하기 위해 브랜드 통합 학습을 유지하기로 결정

**7. LLM 응답 누락 문제**
- 문제: LLM API(GPT-5 mini) 연동 후 빈 응답이 반환되는 현상 발생
- 원인: GPT-5 계열 모델은 답변 생성 전 내부 추론(reasoning) 토큰을
  소비하는 구조라, max_completion_tokens를 낮게(500) 설정하면 추론만
  하다 실제 답변 텍스트를 생성하지 못하고 종료됨
- 해결: max_completion_tokens를 2000으로 상향 조정하여 정상 응답 확인

**8. "내 폰 진단" 가치 하락 계산 오류**
- 문제: 300일 사용 시 겨우 7.7%만 하락하는 비현실적인 결과 발생
- 원인: 감가곡선 함수가 이미 "신품가 대비 남은 가치 비율"을 반환하는데,
  이를 다시 "구매 시점(0일) 값"으로 나누는 이중 계산 오류가 있었음
- 해결: 구매가에 감가비율을 직접 곱하는 방식으로 단순화하여 정상화

**9. "내 폰 진단" → "거래 평가" 개념 전면 수정**
- 문제: 사용자가 저렴하게 구매한 가격을 입력해도, 그 가격 자체가 
  감가 계산의 기준값으로 사용되어 "싸게 산 것"이 반영되지 않고 
  오히려 그 가격에서 추가로 더 깎이는 논리 오류 발견
- 원인: 계산 흐름이 "사용자 구매가 → 감가 → 현재가치"였는데, 
  올바른 흐름은 "신품 출고가 → 감가 → 적정 예상가치" 산출 후 
  "사용자 구매가와 비교"였어야 함
- 추가 발견: 기능을 다시 설계하며, 이 페이지가 기존 "예상가" 페이지와 
  계산 로직이 사실상 동일하다는 것을 확인. 차이는 "비교 대상(실제 
  거래가) 유무"뿐이라는 점을 파악해 페이지명을 "거래 평가"로 변경
- 해결: 신품가 기준 적정가치 산출 → 사용자 거래가와 비교하는 구조로 
  전면 수정, 저장용량 선택 기능 추가, 결과 표시를 "가치 하락률" 대신 
  "가치 비교율(+면 비싸게, -면 싸게 구매)"로 변경

**10. 데이터 소스 전면 교체 결정 — 실제 시장 지식과의 모순 발견**
- 배경: 기존 Kaggle 데이터(2013~2020년 해외 시장) 기반 모델에서, 
  실제 한국 중고시장의 "애플 방어력 강함" 통념과 정반대의 결과(애플이 
  오히려 감가 빠름)가 나온 것을 재검토
- 조사 결과: 인도 중고시장은 "아이폰 희소성 프리미엄"(신품 공급 10% 
  미만 vs 폭발적 중고 수요)으로 방어력이 강하다는 사실 확인, 이는 
  기존 데이터셋(해외/구형 데이터)이 이런 시장 특성을 반영하지 못함을 
  뒷받침하는 근거가 됨
- 결정: Day1 가이드 원문("며칠에 걸친 크롤링은 안 됨")을 재확인한 결과, 
  하루 안에 끝나는 소규모 크롤링은 허용 범위임을 확인하고, 번개장터 
  실시간 API 기반으로 데이터 소스를 전면 교체하기로 결정

**11. 크롤링 데이터 정제 과정의 세부 시행착오**
- 저장용량 정규표현식이 "5G"(통신규격)를 저장용량으로 잘못 인식 
  → 최소 2자리 숫자로 조건 강화하여 해결
- pandas 최신 버전에서 groupby().apply() 사용 시 그룹 기준 컬럼이 
  유실되는 버그 발견 → transform() 방식으로 교체하여 해결
- 검색어를 한글("프로")로만 설정해 영문("Pro") 표기 매물을 놓치는 
  문제 발견 → 모델별 한글/영문/붙여쓰기 등 복수 키워드로 검색해 
  중복 제거 후 병합하는 방식으로 개선

**12. 상태 등급(condition) 분류 로직의 결정적 오류 — "무잔상" 오분류**
- 문제: 등급별 평균가를 비교했을 때 C등급(잔상)이 B등급(일반)과 
  거의 차이가 없는(2%) 비상식적인 결과 발견
- 원인 추적: C등급으로 분류된 매물 제목을 직접 확인한 결과, 
  "무잔상"·"잔상없음"(좋은 상태를 강조하는 표현)이 단순 문자열 
  포함 검사로 인해 "잔상"(나쁜 상태)과 혼동되어 잘못 분류되고 
  있음을 발견
- 해결: "무잔상", "잔상없음"을 먼저 좋은 등급(S)으로 분류하도록 
  조건 순서를 수정. 이 결과 진짜 "잔상 있음" 매물은 781건 중 
  단 1건만 남아, 통계적으로 무의미한 수준임을 확인

**13. 파손/고장 매물의 재포함 및 F등급 신설**
- 배경: 초기에는 파손·고장 매물을 노이즈로 간주해 크롤링 단계에서 
  제외했으나, 실제 거래되는 매물의 일부이므로 별도 등급(F)으로 
  포함하는 것이 더 완성도 높은 서비스라 판단해 정책 변경
- 결과: F등급(6건) 평균가가 신품가 대비 9.6%로, 정상 등급(34~39%)과 
  뚜렷하게 구분되는 것을 확인 → 등급 체계의 실효성 입증

**14. 절대가격 비교의 함정 — "신품가 대비 비율"로 재계산**
- 문제: 등급별 절대가격(원) 평균을 비교했을 때 S/A/B 순서가 
  뒤섞여 나오는 비상식적 결과
- 원인: 각 등급에 우연히 섞여 있는 모델의 가격대 자체가 달라, 
  "등급의 순수한 효과"가 아니라 "그 등급에 어떤 모델이 많은지"가 
  평균을 왜곡시키고 있었음(예: 저가형 최신 모델이 B등급에 
  많이 몰림)
- 해결: 절대가격이 아닌 "실거래가 ÷ 신품가" 비율로 재계산하여, 
  모델 종류 변수를 제거하고 순수 등급 효과만 분리 → S/A > B > F 
  순으로 상식적인 결과 확인

**15. 회귀 모델의 등급 학습 불안정성 — 시장 벤치마크로 대체**
- 문제: 크롤링 데이터(781건)만으로 계산한 등급별 비율은 표본 
  부족(특히 F등급 6건)으로 신뢰도가 낮고, 회귀 모델 역시 등급 
  피처의 중요도가 2% 수준에 불과해 등급 변경 시 예측가가 
  거의 변하지 않거나 순서가 뒤바뀌는 문제 발생
- 해결: 회귀 모델은 브랜드/스펙 기반의 "기준가"(등급 B로 고정) 
  산출만 담당하도록 역할을 제한하고, 실제 등급 반영은 공개된 
  중고폰 시장 벤치마크 수치(S=1.10, A=1.00, B=0.85, F=0.20, 
  A급 기준)로 후처리하는 하이브리드 구조로 전환

**16. 인접 모델 간 예측가 역전 현상 발견**
- 문제: 아이폰16프로(신품가 230만원)의 예측가가 아이폰16프로맥스
  (신품가 250만원)보다 높게 나오는 논리적 모순 발견
- 원인: 두 모델은 스펙이 매우 유사한 데다, 각각 표본이 8건/13건에 
  불과해 회귀 모델이 신품가 순서를 안정적으로 학습하지 못함
- 결론: 인위적 보정 대신 한계로 인정 — 표본이 적은 인접 모델 
  간 순서 오류는 표본을 추가 확보해야 근본적으로 해결 가능하며, 
  화면에도 이 한계를 안내 문구로 명시

  **17. 저장용량 데이터 오염 발견 및 대응**
- 문제: 모델별·용량별 표본 수를 점검하는 과정에서, "521GB", "514GB", 
  "16GB"(아이폰16 기준) 등 실제로 존재하지 않는 저장용량 값이 크롤링 
  데이터에 섞여 있는 것을 발견함
- 원인 추정: 정규표현식이 가격 등 다른 숫자를 저장용량으로 잘못 
  인식했을 가능성이 있음
- 조치: 각 모델의 실제 출시 저장용량(phone_models.py 매핑표) 기준으로 
  존재하지 않는 조합을 필터링하여 재학습에서 제외
- 추가 조치: 그럼에도 표본이 5개 이하인 모델·용량 조합이 다수 
  남아있어(전체의 약 50%), 512GB 이상 옵션 선택 시 "학습 표본이 적어 
  예측 신뢰도가 낮을 수 있다"는 경고 문구를 5개 페이지 전체에 반영함
